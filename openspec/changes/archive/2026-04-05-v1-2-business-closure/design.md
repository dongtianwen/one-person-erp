## Context

数标云管 v1.0 已完成核心业务模块（客户/项目/合同/财务/看板/备份），v1.1 已完成合规留痕能力（提醒/文件索引/财务留痕扩展/默认模板）。v1.2 目标是补全客户→报价→合同的完整业务闭环，新增客户资产托管管理，替换定时任务为事件驱动逾期检查，并落地本地容错日志和备份验证。

**当前技术栈**：FastAPI + SQLAlchemy 2.0 + SQLite + Vue 3 + Element Plus + Alembic

**约束**：
- 本地单机部署，SQLite 数据库
- 所有变更只能新增字段/新建表，禁止修改或删除已有字段
- 适配夜间关机场景（无定时任务可用）

## Goals / Non-Goals

**Goals:**
- 实现完整的客户→报价→合同业务链路
- 管理客户托管资产（服务器/域名/SSL等）
- 事件驱动逾期检查替代定时任务
- 本地容错日志系统
- 备份文件完整性校验

**Non-Goals:**
- 不做报价单 PDF 导出（后续版本考虑）
- 不做自动邮件/消息通知（本地单机场景不需要）
- 不做现金流预测（v1.3 范围）
- 不做增值税发票台账（v1.3 范围）

## Decisions

### D-1: 报价单编号生成策略
**选择**: 复用合同编号生成模式（BJ-YYYYMMDD-序号，当日从 001 起）
**理由**: 与合同编号（HT-YYYYMMDD-序号）保持一致的生成逻辑，复用现有 `generate_contract_number` 工具函数，改前缀即可。SQLite 事务保证原子性。

### D-2: 客户资产嵌入客户详情页 vs 独立页面
**选择**: 在客户详情页新增 Tab，不创建独立导航入口
**理由**: 资产始终属于特定客户，不需要全局独立管理。Tab 方式与现有客户详情页结构一致（已有跟进记录等 Tab），开发量更小，用户操作路径更短。

### D-3: 事件驱动检查触发点设计
**选择**: 服务启动全量检查 + 仪表盘统计接口触发限流检查
**理由**: 适配本地电脑夜间关机场景。启动时全量检查确保长时间关机后数据一致；仪表盘接口是高频访问入口，保证检查及时性。90天窗口 + 1000条上限避免性能问题。
**替代方案**: 前端定时轮询 — 增加不必要的复杂度，且关机时无意义。

### D-4: 日志系统使用 Python 标准库
**选择**: `logging.handlers.RotatingFileHandler`
**理由**: 零外部依赖，满足 PRD 要求。通过 FastAPI middleware 拦截请求/响应，通过 SQLAlchemy event 监听数据库异常。
**替代方案**: loguru — 功能更强但引入外部依赖，标准库已满足需求。

### D-5: 备份验证实现方式
**选择**: SQLite `ATTACH DATABASE` 加载备份文件，查询核心表记录数对比
**理由**: ATTACH 是 SQLite 原生能力，不需要复制文件。对比记录数即可验证完整性，实现简单可靠。
**替代方案**: 文件 MD5 校验 — 只能验证文件完整性，无法验证数据可读性。

### D-6: 一键转合同实现方式
**选择**: 后端提供专门的 `POST /api/v1/quotations/{id}/convert-to-contract` 端点
**理由**: 涉及多步操作（锁定报价单、创建合同草稿、建立关联），需要事务保证。单一端点比前端多步调用更可靠。

## Risks / Trade-offs

- **[启动全量检查耗时]** → 单机 SQLite 数据量有限（<10万条），全量检查预计 <1 秒。超限时有日志记录。
- **[备份验证临时文件残留]** → 使用 try-finally 确保清理，即使异常也不留残留文件。
- **[报价转合同事务一致性]** → 使用 SQLAlchemy session 事务，任一步骤失败整体回滚。
- **[事件驱动检查频率]** → 仪表盘每次调用都触发检查，但 90天窗口+1000条上限控制性能开销。后续可考虑加缓存/节流。
