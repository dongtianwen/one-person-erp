## Context

v1.0~v1.10 已建立完整的报价→合同→项目→财务→发票业务链路。v1.11 在此基础上新增 AI 项目交付的核心追踪能力：数据集→标注→训练→模型→交付→验收全链路。

当前系统已有 acceptances（验收）、deliverables（交付物）、requirements（需求）等表，需要复用而非重建。项目详情页（ProjectDetail.vue）已有 Tab 切换机制，可直接扩展。

核心约束：
- 只做记录、关联、追踪、验收；不做执行、调度、计算
- 版本冻结规则必须在后端强制执行，前端只读只是 UX 提示
- 所有跨表状态更新必须原子事务

## Goals / Non-Goals

**Goals:**
- 完整的数据集版本管理，含冻结/解冻规则
- 标注任务全生命周期追踪，质检结果可查
- 训练实验与数据集版本关联，自动 in_use 标记
- 模型版本管控，交付后冻结
- 交付包管理，deliver 时原子更新模型状态，验收时原子更新包状态
- 全链路可追溯：模型→实验→数据集→标注任务
- 复用 v1.5 表结构，最小化侵入

**Non-Goals:**
- 在线标注平台（仅记录标注任务元数据）
- 模型训练编排（仅记录实验参数和指标）
- 自动评测平台
- 多人协作/GPU 调度
- 文件哈希自动计算
- 自动化 MLOps 流水线

## Decisions

### D1: 复用 acceptances/deliverables/requirements 表而非新建
**选择**: ALTER TABLE 新增列
**替代方案**: 新建独立的 delivery_acceptances 表
**理由**: acceptances 表已有完整的验收记录、付款提醒触发等逻辑。通过 delivery_package_id 外键区分传统验收和交付包验收，复用现有代码路径（提醒触发、notes 追加等），减少重复代码。

### D2: in_use 状态由系统自动设置，API 禁止直接传入
**选择**: link_dataset_version_to_experiment 事务内自动设置
**替代方案**: 允许手动设置 in_use
**理由**: in_use 表示"正在被实验引用"，是关系型推导状态。允许手动设置会导致数据不一致（可能标记 in_use 但实际无引用）。系统自动设置可保证引用计数与状态一致。

### D3: 版本冻结采用白名单字段模式
**选择**: 定义 FROZEN_FIELDS 常量列表，冻结状态下拦截修改
**替代方案**: 全字段冻结 + 哈希校验
**理由**: notes 和 change_summary 需要在任何状态下可编辑（用于补充说明）。白名单模式精确控制哪些字段冻结，灵活且安全。

### D4: 交付链路模型 — 交付包作为聚合根
**选择**: delivery_packages 聚合模型版本和数据集版本，通过 package_model_versions 和 package_dataset_versions 多对多关联
**替代方案**: 交付包只关联模型，模型关联数据集
**理由**: 实际交付场景中，一个交付包可能同时包含模型和配套数据集（如标注规范文档、样本数据）。交付包作为聚合根，deliver 时可原子更新所有关联模型状态。

### D5: 前端新增 5 个 Tab 而非独立页面
**选择**: 在项目详情页扩展 Tab
**替代方案**: 独立路由页面
**理由**: 数据集/标注/实验/模型/交付都是项目维度下的子资源，与现有合同/需求/验收 Tab 并列，用户心智模型一致。复用现有 Tab 切换组件，开发量更小。

### D6: 实验冻结规则 — 关联数据集后冻结
**选择**: experiment_dataset_versions 有关联记录即触发冻结
**替代方案**: 实验开始训练后冻结
**理由**: 一旦实验使用了特定数据集版本，其 project_id/framework/hyperparameters 应不可变，否则实验的可复现性无法保证。解除关联后不自动解冻（因为可能已经产生了模型版本），但允许修改 name/description 等非核心字段。

## Risks / Trade-offs

**[迁移风险] ALTER TABLE 可能与未来版本冲突** → 使用 PRAGMA table_info 检查字段存在性，不存在才 ALTER，幂等执行

**[并发风险] 多个实验同时关联同一数据集版本** → in_use 是引用计数模式，link 时原子事务检查 + UPDATE，unlink 时检查引用数再恢复 ready

**[复杂度风险] 冻结规则跨多表多状态** → 集中定义在 constants.py，utils 函数统一校验，避免散落在各 router 中

**[性能风险] 追溯查询可能涉及多表 JOIN** → 建立必要索引（迁移脚本中已包含），traceability 函数控制查询层级

**[兼容风险] acceptances 表新增 delivery_package_id 必填可能影响旧数据** → 字段设为 NULL 允许，仅在交付包验收场景下必填（通过业务逻辑校验而非 NOT NULL 约束）
