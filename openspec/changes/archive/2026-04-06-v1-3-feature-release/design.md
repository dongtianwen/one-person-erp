## Context

数标云管 v1.2 已实现基础的财务记账（收入/支出 CRUD）、合同管理、报价单、客户资产和提醒功能。后端使用 FastAPI + SQLite，前端 Vue 3 + Element Plus。数据库通过 Alembic 风格的手动迁移脚本管理，每个版本一个迁移文件。

v1.3 在 v1.2 基础上新增三个业务模块：
1. **外包协作**（FR-302）：记录外包费用的合规信息（外包方姓名、发票状态、税务处理）
2. **发票台账**（FR-303）：发票字段精细化管理 + 进销项税额自动计算 + 季度增值税统计
3. **现金流预测**（FR-301）：基于合同应收账款 + 历史支出，生成 90 天按周预测

## Goals / Non-Goals

**Goals:**
- 数据库平滑迁移，新增字段 + 索引，不影响现有数据
- 所有校验规则在创建和更新接口中一致生效
- 税额计算使用 Decimal 类型，精度为 2 位小数
- 核心计算逻辑提取为独立函数，可直接在测试中调用
- 前端动态字段交互：显示/隐藏时值正确清理，提交 payload 不含隐藏字段
- 现金流预测折线图和季度增值税卡片在看板展示

**Non-Goals:**
- 不做权限系统增强
- 不做多币种支持
- 不做发票 OCR 识别或自动导入
- 不做现金流预测的 AI/ML 模型，使用确定性公式
- 不做移动端适配

## Decisions

### 1. 数据库迁移策略：手动迁移脚本 + 快照验证

**选择**：使用 `backend/migrations/v1_3_migrate.py` 手动迁移脚本，迁移前后对数据库做快照对比。

**理由**：项目一直使用手动迁移（v1.1、v1.2 均如此），引入 Alembic 会增加不必要的复杂度。快照验证（行数、抽样记录、索引检查）确保迁移安全性。

**备选**：Alembic 自动迁移 → 否决，单人项目不需要迁移框架的额外维护成本。

### 2. 税额计算：Decimal 类型 + 独立函数

**选择**：`calculate_tax_amount(amount, tax_rate)` 使用 Python `Decimal` 类型，`round(x, 2)` 四舍五入。

**理由**：float 类型在财务计算中有精度丢失风险，Decimal 是 Python 财务应用的标准选择。独立函数便于单元测试和复用。

### 3. 字段校验分层：Pydantic Schema 验证 + 业务逻辑层强制置 NULL

**选择**：
- 外包/发票的必填校验在业务逻辑层实现（返回 HTTP 422）
- 非外包/无发票时，在业务逻辑层强制将相关字段设为 NULL
- Pydantic Schema 中 `tax_amount` 标记为不可写入

**理由**：校验逻辑依赖上下文（如 `category == OUTSOURCING` 才需要校验外包字段），Pydantic 的 field validator 不适合处理跨字段条件校验。

### 4. 现金流预测：确定性公式而非 ML

**选择**：收入预测基于合同 `expected_payment_date` 分配到周，支出预测基于最近 3 个月历史均值。

**理由**：单人企业数据量极少，ML 模型无训练基础。确定性公式可解释、可审计、可精确测试。

### 5. 前端动态字段：Vue 响应式 + payload 清理

**选择**：使用 Vue 3 的 `computed` + `watch` 控制字段显隐，提交前构建 payload 时只包含当前可见字段的值。

**理由**：Element Plus 的表单组件天然支持动态显隐，关键是在提交前清理 payload，不能依赖后端过滤。

### 6. 新增工具文件组织

```
backend/core/
├── constants.py       # 新增常量（已有文件，追加）
├── finance_utils.py   # 新增：税额计算 + 季度统计
└── cashflow_utils.py  # 新增：现金流预测工具函数
```

## Risks / Trade-offs

- **[迁移失败风险]** → 迁移前做完整快照，迁移后 6 项验证，失败时写入 PROGRESS.md 并停止
- **[前端 payload 清理不彻底]** → 约束前端在切换分类时主动清理字段值，而非依赖后端兜底；通过 Network 面板验收
- **[现金流预测精度不足]** → 使用 Decimal + round(x, 2) 保证精度，不接受 float 计算
- **[合同状态白名单遗漏]** → 严格限定为 `["active", "executing"]`，其他状态一律排除
- **[测试回归]** → 每个簇完成后运行全量测试，0 FAILED 才进入下一簇
