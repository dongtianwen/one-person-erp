## Context

### 当前状态
v1.0 ~ v1.6 已完成核心功能模块：客户管理、项目管理、报价管理、合同管理、里程碑管理、需求管理、交付物管理、验收管理、发布管理、变更单（基础框架）、维保期、现金流预警等。

当前系统存在的问题：
- **需求变更无控制**：需求可在任何时间直接修改，无审批流程，导致范围蔓延
- **收款脱节**：里程碑完成与收款状态无关联，应收账款跟踪困难
- **项目关闭随意**：无强制条件检查，可能存在未完成交付即关闭的风险
- **工时数据缺失**：无工时记录，无法评估项目健康度和改进报价精度

### 约束条件
- SQLite 数据库（不支持 ADD COLUMN IF NOT EXISTS，迁移脚本需检查字段是否存在）
- Python 后端 + Streamlit 前端
- 必须保持 v1.0 ~ v1.6 全量测试通过（0 FAILED）
- 不引入新的外部依赖

## Goals / Non-Goals

**Goals:**
1. 建立需求冻结机制，报价确认后所有变更需经客户确认
2. 里程碑与收款节点强绑定，确保收款状态可追溯
3. 项目关闭前强制校验所有条件，防止 premature closure
4. 记录工时数据，计算偏差率，超过阈值强制填写原因
5. WIP 看板展示进行中项目，超过警戒线显示警告

**Non-Goals:**
- 自动报价推荐（未来版本）
- 组件沉淀库（未来版本）
- WIP 硬性拦截（WIP 仅作展示，不拦截任何操作）
- AI 自动定价（未来版本）

## Decisions

### 1. 变更冻结触发点选择

**决策**：以 `quotation.status = accepted` 作为需求冻结触发点。

**理由**：
- 报价确认标志着项目范围已商定，此时冻结需求符合业务逻辑
- 与现有 quotation 流程无缝衔接，无需新增状态
- 客户确认报价意味着对范围达成一致，后续变更应走正式流程

**备选方案**：
- 以合同签订为触发点 → 考虑到报价后可能存在合同谈判期，需求冻结应更早生效
- 以项目启动为触发点 → 过晚，报价到启动期间的需求变更无法控制

### 2. requirement_changes 与 change_orders 职责分离

**决策**：两张表严格区分职责：
- `requirement_changes`：需求字段的内部历史快照（字段级变更记录），每次修改需求时自动写入
- `change_orders`：正式变更申请，包含额外工时与费用，需客户确认

**理由**：
- 历史快照与计费变更分开，职责清晰
- `requirement_changes` 用于内部审计，`change_orders` 用于对客计费
- 需求冻结前后均写 `requirement_changes`，保证审计完整性

### 3. final_acceptance_passed 判定来源

**决策**：严格以 `acceptances` 表中 `acceptance_type = 'final'` 的最新记录的 `status = 'passed'` 为准。

**理由**：
- 验收表是项目验收的唯一权威来源
- 取最新记录支持重验收场景（首次失败后再次验收）
- 禁止用里程碑状态或交付物状态代替，确保判定逻辑单一

### 4. WIP 限制的性质

**决策**：WIP 仅作运营提示，不参与任何状态流转校验。

**理由**：
- WIP 是运营管理工具，不应成为业务硬约束
- 紧急情况下需要能够并行推进多个项目
- 前端展示警告即可，由用户自行判断

**实现**：
- `WIP_DISPLAY_LIMIT` 常量仅供前端看板使用
- 禁止在后端校验逻辑中引用此常量

### 5. 工时偏差阈值判断来源

**决策**：后端是阈值的唯一判断来源，通过 `deviation_exceeds_threshold` 字段返回前端。

**理由**：
- 阈值可能需要调整（20% → 其他值），后端统一修改
- 前端不得自行计算，避免阈值不同步
- 接口驱动前端行为，保持后端权威性

**实现**：
- `WORK_HOUR_DEVIATION_THRESHOLD = 0.20` 定义在 `backend/core/constants.py`
- 后端计算 `deviation_rate` 和 `deviation_exceeds_threshold`
- 前端根据 `deviation_exceeds_threshold` 决定是否必填偏差备注

### 6. 变更单状态流转设计

**决策**：状态流转为 `pending` → `confirmed / rejected / cancelled`，除 `confirmed` 外三个状态为终态。

**理由**：
- `pending` 是初始状态，等待客户确认
- `confirmed` 后变更落库，进入开发
- `rejected` 和 `cancelled` 是终态，不允许再流转
- `confirmed` 理论上可以再次修改（如客户反悔），但本版本不实现

### 7. 里程碑收款状态流转

**决策**：`payment_status` 流转 `unpaid` → `invoiced` → `received`，且 `invoiced` 需要里程碑 `status=completed`。

**理由**：
- 只有已完成的里程碑才能开票，符合业务逻辑
- `received` 是终态，款项到账后不再变更
- 非 completed 里程碑触发 `invoiced` 返回 HTTP 409

### 8. 项目关闭条件设计

**决策**：四个条件全部满足才允许关闭，原子事务执行。

**理由**：
- `all_milestones_completed`：确保所有工作完成
- `final_acceptance_passed`：确保客户验收通过
- `payment_cleared`：确保款项结清
- `deliverables_archived`：确保交付物已归档

**实现**：
- `close-check` 接口只读，返回每项条件状态
- `close` 接口执行前重新校验，不信任前端状态
- 关闭成功后写入 `close_checklist` JSON 快照

### 9. 数据库迁移策略

**决策**：迁移脚本使用 `PRAGMA table_info` 检查字段是否存在，不存在才执行 `ALTER TABLE`。

**理由**：
- SQLite 不支持 `ADD COLUMN IF NOT EXISTS`
- 避免因字段已存在而报错中止
- 支持重复执行迁移脚本

### 10. 事务一致性保证

**决策**：以下操作必须单一事务，失败全部回滚：
- 变更单状态确认（含 `requirement_changes` 快照写入）
- 里程碑收款状态流转
- 项目关闭（条件校验 + 状态更新 + 快照写入）

**理由**：
- 保证数据一致性
- 避免部分成功部分失败的状态

### 11. 函数复杂度控制

**决策**：所有函数 ≤ 50 行，圈复杂度 < 10，超出者必须拆分。

**理由**：
- 提高代码可读性和可维护性
- 便于单元测试

## Risks / Trade-offs

### 风险 1：需求冻结影响现有工作流

**描述**：需求冻结后，任何变更都需要走 change_orders 流程，可能影响效率。

**缓解措施**：
- 提供 change_orders 的便捷 API，减少操作复杂度
- 前端提供清晰的提示和引导
- 确保变更单流程顺畅，避免卡顿

### 风险 2：项目关闭条件过于严格

**描述**：四个条件全部满足才能关闭，可能导致项目无法及时关闭。

**缓解措施**：
- `close-check` 接口返回详细的未满足项，方便用户了解情况
- 提供 `close_checklist` 字段，记录关闭时的状态快照
- 考虑未来版本增加管理员强制关闭功能

### 风险 3：工时偏差阈值硬编码

**描述**：20% 的阈值硬编码在后端，调整需要重新部署。

**缓解措施**：
- 注释中说明阈值的业务含义
- 未来版本可考虑配置化（数据库或环境变量）

### 风险 4：数据库迁移失败

**描述**：SQLite 不支持回滚 DDL 操作，迁移失败后可能无法恢复。

**缓解措施**：
- 迁移前记录快照（行数、字段、抽样记录）
- 迁移后验证所有变更
- 提供迁移测试，确保迁移脚本的正确性

### 风险 5：final_acceptance 记录缺失

**描述**：如果项目中没有 `acceptance_type = 'final'` 的记录，`final_acceptance_passed` 判定为 false，项目无法关闭。

**缓解措施**：
- `close-check` 接口明确返回该条件未满足
- 前端显示清晰的提示
- 引导用户创建最终验收记录
