## Context

v1.0~v1.9 构建了完整的 ERP 业务系统（客户、合同、项目、报价、发票、财务、风控）。当前错误处理仅返回 `{"detail": "...", "code": "..."}` 格式，用户看到一句话提示后无法理解原因和下一步操作。新用户对业务流程顺序不熟悉，关键字段含义不明确。

现有架构：
- 后端 FastAPI + SQLAlchemy，异常通过 `exception_handlers.py` 统一处理
- 前端 Vue 3 + Element Plus，axios 拦截器统一处理错误响应
- 帮助内容当前完全缺失

## Goals / Non-Goals

**Goals:**
- 在用户操作出错时提供可执行的原因解释和下一步引导
- 提供业务流程总览帮助新用户理解操作顺序
- 为关键字段提供 inline 提示减少填写错误
- 每个主要页面提供按需查看的帮助抽屉
- 所有帮助内容随代码版本管理，零数据库依赖

**Non-Goals:**
- 不建帮助内容数据库表
- 不建帮助内容查询 API（无 GET /help/*）
- 不做独立知识库 / 帮助站
- 不做 AI 问答机器人
- 不做帮助内容后台管理界面
- 不做全文搜索

## Decisions

### D1: 帮助内容存储方式 — 纯静态文件

**选择**：后端 `backend/core/help_content.py`（Python 字典），前端 `frontend/src/constants/help.js`（JS 对象）

**替代方案**：数据库表 + API 接口 → 被否决

**理由**：
- 帮助内容变更频率低，随代码版本管理更可靠
- 无需建表、写 CRUD、写 API，减少开发和维护成本
- 前端直接导入 JS 文件，无需网络请求，无延迟
- 部署即生效，无需数据迁移

### D2: 错误帮助注入方式 — 只追加 help 字段

**选择**：在现有 `{"detail": "...", "code": "..."}` 基础上追加 `"help": {...}` 字段

**规则**：
- 禁止修改 detail 和 code 内容
- 无对应帮助内容的错误码不返回 help 字段
- HTTP 5xx 不附加 help

**理由**：向后兼容，现有前端断言不受影响，渐进增强

### D3: 前端错误展示 — 全局拦截器统一处理

**选择**：修改现有 axios 响应拦截器，根据 help 字段决定展示方式

**规则**：
- 4xx 无 help → 保持现有 ElMessage.error(detail)
- 4xx 有 help → 弹出 ErrorHelp 对话框（原因 + 步骤列表）
- 5xx → ElMessage.error("服务器错误，请稍后重试")

**理由**：全局处理避免每个页面重复实现，统一用户体验

### D4: 帮助内容缺失降级 — 静默处理

**选择**：help.js 无对应 key 时，不渲染、不报错、不白屏

**理由**：帮助系统故障不得影响主业务功能，降级为无帮助状态

### D5: 错误码注册表 — 集中管理

**选择**：新建 `backend/core/error_codes.py` 集中定义所有错误码

**理由**：
- 避免错误码散落在各处难以维护
- 便于与 help_content.py 保持同步
- 后续版本新增错误码时有明确的注册入口

## Risks / Trade-offs

- **[帮助内容过时风险]** → 帮助内容随代码版本管理，每次修改业务逻辑时同步更新 help_content.py 和 help.js
- **[错误码与帮助映射不同步]** → 簇 F 强制检查：error_codes.py 中每个错误码必须评审是否需要 help 映射，未映射的须注释原因
- **[前端文件体积增长]** → help.js 为纯文本数据，预估总量 < 10KB，不影响性能
- **[FieldTip 组件覆盖不全]** → PRD 明确列出必须覆盖的字段清单，簇 C 验收时逐项检查
