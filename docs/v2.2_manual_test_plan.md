# v2.2 手动测试计划

## 前置准备

1. 启动后端：`cd backend && uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload`
2. 启动前端（如有）：`cd frontend && npm run dev`
3. 打开浏览器访问 `http://localhost:5173`（前端）或 `http://localhost:8000/docs`（Swagger API 文档）
4. 用 admin / admin123 登录获取 token

---

## Cluster A1 — snapshot 底座（6 项）

### A1-1 报告快照写入
- **操作**：在 Swagger 中 POST `/api/v1/reports/generate`，传入 `report_type=report_project`, `entity_id=1`（需先有项目数据）
- **预期**：返回 success，数据库 `entity_snapshots` 表新增一条 `entity_type='report'` 记录
- **验证**：查 SQLite 数据库 `SELECT * FROM entity_snapshots WHERE entity_type='report'`

### A1-2 纪要快照写入
- **操作**：POST `/api/v1/minutes`，body：
  ```json
  {"title": "测试纪要", "participants": "张三,李四", "conclusions": "结论A", "action_items": "待办B", "risk_points": "风险C", "client_id": 1}
  ```
- **预期**：201/200，`entity_snapshots` 新增一条 `entity_type='minutes'`

### A1-3 模板快照写入
- **操作**：POST `/api/v1/templates/?name=手动测试模板&template_type=delivery&content={{project_name}}交付报告`
- **预期**：201/200，`entity_snapshots` 新增一条 `entity_type='template'`

### A1-4 is_latest 自动切换
- **操作**：对同一实体连续保存两次纪要（PUT 更新）
- **预期**：`entity_snapshots` 表中该实体有 2 条记录，旧版 `is_latest=0`，新版 `is_latest=1`

### A1-5 version_no 单调递增
- **操作**：同一实体保存 3 次
- **预期**：version_no 分别为 1、2、3

### A1-6 版本对比 API
- **操作**：GET `/api/v1/reports/{id}/diff?version_a=1&version_b=2`（或直接用 snapshot_service 测试）
- **预期**：返回两个版本的 content JSON

## Cluster D — 工具入口台账 + 客户线索台账（9 项）

### D-1 工具入口 CRUD
- **操作**：
  - POST `/api/v1/tool-entries` body: `{"action_name": "数据标注", "tool_name": "Label Studio"}`
  - GET `/api/v1/tool-entries` 确认列表可见
  - DELETE `/api/v1/tool-entries/{id}`
- **预期**：全流程正常

### D-2 工具入口状态流转
- **操作**：
  - 创建 entry（status=pending）
  - PATCH `/api/v1/tool-entries/{id}/status` body: `{"status": "in_progress"}`
  - PATCH 同上 body: `{"status": "done"}`
  - PATCH 同上 body: `{"status": "backfilled", "is_backfilled": true}`
- **预期**：pending→in_progress→done→backfilled 均成功

### D-3 工具入口 is_backfilled 标记
- **操作**：PATCH status 时传入 `{"is_backfilled": true}`
- **预期**：返回 `is_backfilled: true`

### D-4 工具入口无效流转拒绝
- **操作**：从 pending 直接 patch 到 backfilled
- **预期**：400 错误，"TOOL_ENTRY_INVALID_TRANSITION"

### D-5 工具入口按状态过滤
- **操作**：GET `/api/v1/tool-entries?status=pending`
- **预期**：只返回 pending 状态的记录

### D-6 客户线索 CRUD
- **操作**：
  - POST `/api/v1/leads` body: `{"source": "referral", "status": "initial_contact", "next_action": "电话沟通"}`
  - GET `/api/v1/leads` 确认可见
  - PUT `/api/v1/leads/{id}` body: `{"status": "intent_confirmed"}`
  - DELETE `/api/v1/leads/{id}`
- **预期**：全流程正常

### D-7 线索状态流转
- **操作**：initial_contact → intent_confirmed → converted
- **预期**：每步均成功

### D-8 线索无效流转拒绝
- **操作**：从 initial_contact 直接转为 converted
- **预期**：400 错误，"LEAD_INVALID_TRANSITION"

### D-9 线索转化关联 client_id
- **操作**：PUT status=converted 时同时传 client_id=1
- **预期**：返回 `client_id: 1`

### D-10 leads 不重复 clients 字段
- **检查**：leads 表只有 source/status/next_action/client_id/project_id/notes 六字段
- **预期**：无 name/contact_phone/email 等 clients 已有字段
