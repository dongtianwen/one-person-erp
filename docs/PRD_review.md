## 🤖 Agent 技能定义

### Phase 1: 编写测试用例（Red）
**Agent 角色**：测试工程师（Test Engineer）

**技能要求**：
- 根据需求编写详细的测试用例
- 测试必须覆盖正常的主干流程以及关键的边界 / 异常情况

**约束条件**：
- ⚠️ 此阶段禁止编写任何业务代码
- ⚠️ 每个测试用例必须对应一个明确的需求条目（FR-XXX）

**需覆盖的核心业务规则（必须全部转化为测试用例）**：

认证模块：
- 密码错误 5 次后账户锁定 30 分钟
- Access Token 有效期 30 分钟，Refresh Token 有效期 7 天
- 登出后 Token 失效，无法访问受保护接口

客户管理模块：
- 同一公司 + 联系人组合视为重复客户，新增时必须提示
- 客户状态变更为"流失"时，必须填写流失原因，否则拒绝变更
- 有关联项目或合同的客户，禁止删除（软删除也需拦截）
- 客户状态变更为"成交"**不触发任何自动操作**（已修正 PRD 原规则）

项目管理模块：
- 项目开始日期不能晚于结束日期
- 所有里程碑完成后，项目进度自动设为 100%
- 项目预算变更必须记录变更原因
- 删除项目时，级联删除关联任务和里程碑

合同管理模块：
- 合同编号自动生成，格式严格为 HT-YYYYMMDD-序号（当日序号从 001 起）
- 合同状态只能按顺序流转：草稿 → 生效 → 执行中 → 已完成（不可跳跃）
- 合同终止时，必须填写终止原因
- 合同金额变更后，同步更新关联项目的预算字段

财务管理模块：
- 收入记录**可选**关联合同（已修正 PRD 原规则，不强制）
- 已确认的财务记录**可修改**，但必须自动记录修改前后的值到变更日志表（已修正 PRD 原冲销规则）
- 发票号码全局唯一校验
- 月度报表计算：净利润 = 本月总收入 - 本月总支出

数据看板模块：
- 各指标计算公式正确性验证（客户转化率、项目延期率、应收账款）
- 看板任意单项图表数据获取失败时，其余图表不受影响，继续渲染

备份功能：
- 调用备份接口后，SQLite 数据库文件被成功复制到指定目录
- 备份文件名包含时间戳，不覆盖历史备份

**交付物**：
测试目录与文件（`tests/test_auth.py`、`tests/test_customer.py`、`tests/test_project.py`、`tests/test_contract.py`、`tests/test_finance.py`、`tests/test_dashboard.py`、`tests/test_backup.py`）。所有测试用例在此阶段运行必然失败（无业务代码），这是符合预期的。输出时需列出测试用例清单（函数名 + 对应需求编号）。

**检查点**：
- ✅ 每个核心功能是否至少有一个对应的测试用例？
- ✅ 上述所有业务规则是否全部转化为了可执行的测试断言？
- ✅ 是否包含边界条件和异常情况的测试（如状态非法流转、重复客户、锁定账户）？
- ✅ 运行测试后是否全部显示 FAILED（符合预期）？


### Phase 2: 实现功能代码（Green）
**Agent 角色**：开发工程师（Developer）

**技术选型（严格遵守，不得自行更换）**：
- 后端：Python / FastAPI / SQLAlchemy 2.0（ORM，参数化查询防 SQL 注入）
- 前端：Vue 3 / Element Plus
- 数据库：SQLite（本地单机部署）
- 测试框架：pytest
- 密码加密：bcrypt
- 认证：JWT（python-jose 或 PyJWT）
- 代码规范：后端 PEP 8 + 强制类型注解，前端 ESLint

**架构要求**：
- 前后端分离：FastAPI 提供 REST API，Vue 3 独立前端项目
- 必须生成一键启动脚本（Windows 用 `start.bat`，Mac/Linux 用 `start.sh`），同时启动后端和前端服务
- API 文档：FastAPI 自动生成 Swagger（`/docs`），保持更新
- 敏感配置（SECRET_KEY 等）必须通过 `.env` 文件管理，禁止硬编码

**异常处理策略（混合策略，强制执行）**：
- 核心模块（认证 / 合同 / 财务）：快速失败——出错立即抛出异常，记录结构化日志，前端显示友好中文提示，不暴露技术细节
- 展示模块（看板 / 统计图表）：静默降级——单项数据获取失败时，该组件显示"数据加载失败，请刷新"，不影响页面其他部分渲染

**约束条件（最关键，强制执行）**：
- ⚠️ **最小实现原则**：每次只实现能让**当前一个**失败测试变为 Pass 的最少代码。
- ⚠️ 每实现一个测试的通过后，必须立即停止，输出以下格式的报告，再等待下一步指令：
  ```
  ✅ [test_函数名] 已通过
  剩余失败测试：[数量] 个
  等待指令继续执行下一个测试的实现。
  ```
- ⚠️ 严禁一次性实现多个测试的逻辑。
- ⚠️ 严禁在测试未覆盖的地方添加任何冗余逻辑。

**模块扩展性要求**：
- 工单系统、文件管理、消息通知（v1.1 计划功能）在架构上预留接入点，但 v1.0 不实现
- 保持模块间低耦合，各模块通过明确定义的服务层接口通信

**交付物**：
业务逻辑代码，分批次提交，每次提交确保运行对应测试用例全部 Pass。

**检查点**：
- ✅ 当前实现是否只覆盖了已有测试的范围？
- ✅ 运行全部测试是否无新增的 FAILED？
- ✅ 异常处理是否按混合策略分类执行？


### Phase 3: 重构优化（Refactor）
**Agent 角色**：代码审查员（Code Reviewer）

**任务**：
- 在保证测试全部通过的前提下，优化代码结构
- 消除魔术数字、提取重复代码、优化命名
- 确保代码符合工程规范（函数 ≤ 50 行，圈复杂度 < 10）
- 验证异常处理策略已按混合策略正确落地（核心模块快速失败、展示模块静默降级）
- 确认所有已修正的业务规则已正确实现（收入可选关联合同、财务记录可编辑带变更日志、无"成交自动关联合同"逻辑、内置备份功能）

**约束条件**：
- ⚠️ 每次重构后必须重新运行全部测试，确认全部 Pass 后才能继续
- ⚠️ 禁止在重构阶段新增功能

**交付物**：
重构后的完整代码 + 运行测试全部 Pass 的输出记录

**检查点**：
- ✅ 重构后所有测试是否仍然全部 Pass？
- ✅ 代码是否消除了重复逻辑？
- ✅ 函数长度和复杂度是否符合规范？
