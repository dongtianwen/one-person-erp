# v1.11 手工测试计划 — 数据标注与模型开发交付台账

## 前置条件

1. 启动后端: `cd backend && python main.py`
2. 启动前端: `cd frontend && npm run dev`
3. 登录系统，进入「项目管理」页面

## 一、数据集台账（Tab: 数据集）

### TC-DS-01 创建数据集

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 进入项目详情 → 数据集 Tab | 显示空列表，有「新建数据集」按钮 |
| 2 | 点击「新建数据集」，填写名称、类型、描述 | 创建成功，列表显示新数据集 |
| 3 | 不填名称提交 | 前端校验提示"请输入名称" |

### TC-DS-02 创建数据集版本

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 展开数据集行，点击「新建版本」 | 弹出版本创建对话框 |
| 2 | 填写 version_no=v1.0, sample_count=500, data_source="公开数据集", label_schema_version="v2" | 创建成功，版本状态=draft |
| 3 | 再创建一个同号版本 v1.0 | 返回 409 冲突 |
| 4 | 填写 version_no=1.0（不含 v 前缀） | 返回校验失败 |

### TC-DS-03 版本冻结字段保护

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 将版本 v1.0 状态改为 ready | 成功，版本标记为 ready |
| 2 | 尝试修改 sample_count 为 600 | 返回 409 DATASET_VERSION_FROZEN |
| 3 | 尝试修改 version_no | 返回 409 |
| 4 | 修改 notes="添加备注" | 成功 |
| 5 | 修改 change_summary="更新说明" | 成功 |

### TC-DS-04 in_use 状态保护

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 创建训练实验并关联此版本 | 版本状态自动变为 in_use |
| 2 | 尝试通过 API 直接将某版本设为 in_use | 返回 422 |
| 3 | 尝试删除 in_use 版本 | 返回 409 VERSION_IN_USE_CANNOT_DELETE |
| 4 | 解除实验关联 | 版本恢复为 ready（无其他引用时） |

### TC-DS-05 版本状态转换

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | draft → 点击「就绪」 | 状态变为 ready |
| 2 | ready → 点击「归档」 | 状态变为 archived |
| 3 | archived 版本核心字段只读 | 修改冻结字段返回 409 |

### TC-DS-06 元数据展示

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 查看版本详情 | 正确显示 data_source / label_schema_version / change_summary |

## 二、标注任务（Tab: 标注任务）

### TC-AT-01 创建标注任务

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 点击「新建标注任务」 | 弹出创建对话框 |
| 2 | 填写名称、选择数据集版本 | 创建成功，状态=pending |
| 3 | 不选择数据集版本提交 | 返回校验失败 |

### TC-AT-02 状态流转

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | pending → 点击「开始」 | 状态变为 in_progress |
| 2 | in_progress → 点击「提交质检」 | 状态变为 quality_check |
| 3 | quality_check → 点击「通过」 | 状态变为 completed |
| 4 | completed 状态下点击任何转换按钮 | 无操作（终态） |

### TC-AT-03 返工流程

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | quality_check → 点击「返工」 | 弹出返工原因对话框 |
| 2 | 不填原因提交 | 前端校验阻止提交 |
| 3 | 填写原因"标注不一致"提交 | 状态变为 rework，rework_reason 已记录 |
| 4 | rework → 点击「重新开始」 | 状态变为 in_progress |

### TC-AT-04 标注规范管理

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 在任务详情中点击「添加规范」 | 弹出规范创建对话框 |
| 2 | 填写规范标题和内容 | 创建成功，规范出现在列表中 |
| 3 | 规范存储在 requirements 表 | 后端确认 requirement_type=annotation_spec, annotation_task_id 已填充 |
| 4 | annotation_tasks 表不包含规范内容 | 数据库检查无规范字段 |

### TC-AT-05 取消任务

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | pending → 点击「取消」 | 状态变为 cancelled（终态） |

## 七、异常与边界测试

### TC-EX-01 无效版本号

| 输入 | 预期 |
|------|------|
| 数据集版本: "1.0" | 拒绝（缺 v 前缀） |
| 数据集版本: "v1" | 拒绝（缺小版本） |
| 数据集版本: "v1.0" | 接受 |
| 模型版本: "v1.0" | 拒绝（需三位） |
| 模型版本: "v1.0.0" | 接受 |

### TC-EX-02 无效状态转换

| 操作 | 预期 |
|------|------|
| 标注: pending → completed | 拒绝（跳过中间状态） |
| 标注: pending → rework | 拒绝（需先到 quality_check） |
| 交付包: draft → delivered | 拒绝（需先 ready） |
| 交付包: draft → accepted | 拒绝 |

### TC-EX-03 并发关联

| 操作 | 预期 |
|------|------|
| 两个实验同时关联同一数据集版本 | 都成功，版本 in_use |
| 解除一个实验关联 | 版本仍 in_use（另一实验仍引用） |
| 解除全部实验关联 | 版本恢复 ready |
