## 三、全版本非功能需求（NFR）

> ⚠️ 以下 NFR 贯穿所有版本，AI 在每个版本开发时必须全部遵守。

### NFR-PERF：性能要求

| 编号 | 要求 | 说明 |
|------|------|------|
| NFR-PERF-001 | API 响应时间 P95 < 500ms | |
| NFR-PERF-002 | 页面首屏加载 < 2 秒 | |
| NFR-PERF-003 | 10 万条数据查询响应 < 1 秒 | 前提：索引已按 NFR-DB-001 建立 |

### NFR-DB：数据库索引策略（必须）

> ⚠️ AI 生成代码默认不建索引，以下索引必须在建表时显式创建，否则数据量增长后查询性能将严重下降。

| 编号 | 表名 | 必须建立索引的字段 |
|------|------|------------------|
| NFR-DB-001 | reminders | reminder_date, status |
| NFR-DB-002 | finance_records | date, funding_source |
| NFR-DB-003 | file_indexes | expiry_date |
| NFR-DB-004 | quotations（v1.2）| status, expiry_date |
| NFR-DB-005 | customer_assets（v1.2）| expiry_date, customer_id |

### NFR-SEC：安全要求

| 要求 | 说明 |
|------|------|
| 密码存储 | bcrypt 哈希，禁止明文 |
| 敏感配置 | .env 环境变量，禁止硬编码 |
| 资产账号信息 | 只记录账号名，严禁存储密码 |
| SQL 注入防护 | SQLAlchemy ORM 参数化查询 |

### NFR-LOG：本地容错日志（v1.2 起强制）

| 要求 | 说明 |
|------|------|
| 实现方式 | Python 标准库 RotatingFileHandler |
| 存放位置 | 项目根目录 logs/ 文件夹 |
| 轮转策略 | 单文件最大 10MB，保留最近 30 个文件 |
| 日志格式 | 时间戳 \| 级别 \| 模块 \| 消息 |
| 强制记录范围 | 所有数据库写入异常 / 文件 I/O 异常 / 事件驱动执行记录 / 系统启停事件 |
| 禁止静默失败 | 任何涉及数据库写入或文件 I/O 的异常，必须写入日志并向前端返回明确错误提示 |
| 禁止记录敏感信息 | 密码、Token、API Key 一律不得写入日志 |

### NFR-DATA：数据完整性

| 要求 | 说明 |
|------|------|
| 数据库变更原则 | 任何变更只能新增字段或新建表，禁止修改或删除已有字段 |
| 历史数据 | 每次版本升级后，历史记录总数必须与升级前一致（迁移脚本验证） |

## 四、v1.1 详细功能说明（开发中）

### 模块一：提醒管理中心

**功能描述**：统一管理所有时间敏感的提醒事项。区分"关键提醒"和"普通提醒"，对关键提醒施加强制性约束。

**功能清单**：

| 功能 | 详细说明 | 验收标准 |
|------|----------|----------|
| 提醒列表 | 展示所有提醒，支持按类型和状态筛选 | 逾期提醒置顶标红，其余按到期时间升序 |
| 新增手动提醒 | 用户手动创建自定义提醒 | 填写标题、类型、提醒日期即可创建 |
| 编辑提醒 | 修改提醒内容和日期 | 已完成提醒不可编辑；关键提醒不可删除 |
| 标记完成 | 将提醒标记为已处理 | 标记后移入"已完成"分组，保留记录 |
| 逾期升级 | 到期日过后未完成的提醒自动变更状态 | 状态变为"已逾期"，仪表盘置顶标红 |
| 仪表盘聚合 | 在仪表盘待办区展示近期未完成提醒 | 逾期优先置顶，其次展示 7 天内到期，最多 10 条 |
| 提醒配置 | 设置页配置年报和税期提醒的周期和截止日 | 配置保存后下一周期自动生效 |

**提醒类型定义**：

| 类型 | 是否关键提醒 | 生成方式 | 触发规则 | 默认配置 |
|------|------------|----------|----------|----------|
| 年报 / 审计提醒 | ✅ 是 | 系统周期生成 | 每年固定日期 | 每年 3 月 31 日 |
| 工商年报提醒 | ✅ 是 | 系统周期生成 | 每年固定日期 | 每年 5 月 31 日 |
| 税期申报提醒 | ✅ 是 | 系统周期生成 | 按申报周期 | 月度，每月 15 日 |
| 合同到期提醒 | ❌ 否 | 系统自动生成 | 到期前 7 天 | 不可配置 |
| 任务截止提醒 | ❌ 否 | 系统自动生成 | 截止前 3 天 | 不可配置 |
| 文件到期提醒 | ❌ 否 | 系统自动生成 | 到期前 30 天 | 不可配置 |
| 自定义提醒 | ❌ 否 | 手动创建 | 用户指定日期 | 自由配置 |

**数据字段**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 提醒标题 | 文本 | ✓ | 最多 200 字符 |
| 提醒类型 | 枚举 | ✓ | 见上方类型定义 |
| 是否关键提醒 | 布尔 | ✓ | 系统自动判断，不可手动修改 |
| 提醒日期 | 日期 | ✓ | 到期或需处理的日期 |
| 状态 | 枚举 | ✓ | 待处理 / 已逾期 / 已完成 |
| 关联对象类型 | 枚举 | | 合同 / 任务 / 文件索引 / 无 |
| 关联对象 ID | 外键 | | 对应关联对象的 ID |
| 备注 | 长文本 | | 最多 1000 字符 |
| 完成时间 | 时间戳 | | 标记完成时自动记录 |
| 生成方式 | 枚举 | | 手动 / 系统自动 |

**业务规则**：
- R-001：年报/税期/工商年报类关键提醒不允许删除，未完成时持续显示仪表盘，无法通过任何操作隐藏
- R-002：逾期检查触发时（见 v1.2 FR-103），所有"待处理"状态过期提醒自动变更为"已逾期"，仪表盘置顶标红
- R-003：同一合同/任务/文件在同一周期内不重复生成提醒（幂等保护）
- R-004：关联对象被删除时，对应系统自动提醒同步删除；关键提醒不受此规则影响

class ReminderType(str, Enum):

### 模块二：文件资料索引

**功能描述**：管理公司重要文件的结构化元数据。不存储文件本身，只记录文件信息、版本、存放位置及关联业务对象。通过 file_group_id 管理同一文件的多个版本，名称字段仅用于展示。

**功能清单**：

| 功能 | 详细说明 | 验收标准 |
|------|----------|----------|
| 新增文件索引 | 录入文件元数据 | 必填项校验，文件名称不可为空 |
| 文件列表 | 展示所有文件，支持筛选和搜索 | 按类型筛选，按文件名搜索，同组版本可展开收起 |
| 编辑文件索引 | 修改文件元数据 | 修改成功后列表即时更新 |
| 删除文件索引 | 删除记录 | 二次确认，同步删除关联提醒 |
| 新增版本 | 为已有文件新增新版本记录 | 共享同一 file_group_id，新版本自动标记为当前有效版本 |
| 到期高亮 | 即将到期的文件高亮显示 | 到期前 30 天标记"即将到期"，过期标红 |

**数据字段**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| file_group_id | UUID | ✓ | 同一文件所有版本共享同一 group_id；新建文件时自动生成；新增版本时复用已有 group_id |
| 文件名称 | 文本 | ✓ | 最多 200 字符，仅用于展示和搜索，不作为分组依据 |
| 文件类型 | 枚举 | ✓ | 见下方枚举定义 |
| 版本号 | 文本 | | 最多 50 字符，如"v1"、"2026版"、"第二次修订" |
| 是否当前有效版本 | 布尔 | ✓ | 同一 file_group_id 内仅一个版本为 true |
| 签署 / 签发日期 | 日期 | | |
| 到期日期 | 日期 | | 填写后自动触发到期提醒 |
| 实际存放位置 | 文本 | | 最多 500 字符，如"D:/公司文件/证照/（已同步至NAS）" |
| 关联对象类型 | 枚举 | | 合同 / 项目 / 财务记录 / 无 |
| 关联对象 ID | 外键 | | 对应关联对象的 ID |
| 经办机构 | 文本 | | 最多 200 字符 |
| 备注 | 长文本 | | 最多 1000 字符 |

```python
class FundingSource(str, Enum):
    COMPANY_ACCOUNT = "company_account"      # 对公账户
    PERSONAL_ADVANCE = "personal_advance"    # 个人垫付
    REIMBURSEMENT = "reimbursement"          # 报销
    LOAN = "loan"                            # 借款
    LOAN_REPAYMENT = "loan_repayment"        # 借款归还
    OTHER = "other"                          # 其他

class SettlementStatus(str, Enum):
    OPEN = "open"          # 未结清（垫付/借款尚未归还）
    PARTIAL = "partial"    # 部分归还
    CLOSED = "closed"      # 已结清
```

**结算状态使用规则**：
- 仅当资金来源为"个人垫付"或"借款"时，结算状态字段显示且必填
- 其他资金来源的记录，结算状态字段隐藏且不存储
- 当关联记录中包含对应的报销/归还记录时，系统提示用户手动更新结算状态（不自动变更，避免逻辑误判）

**关联记录使用场景**：

| 场景 | 操作方式 |
|------|----------|
| 个人垫付后报销 | 报销记录 → 关联原垫付记录，原垫付记录结算状态更新为 closed |
| 借款后归还 | 归还记录（资金来源=借款归还）→ 关联原借款记录，原借款记录结算状态更新为 closed |
| 借款部分归还 | 归还记录关联原借款记录，原借款记录结算状态更新为 partial |

**新增业务规则**：
- FIN-006：支出记录资金来源必填，未填时提交被拦截
- FIN-007：资金来源为"个人垫付"或"借款"时，业务说明和结算状态均必填
- FIN-008：填写关联记录 ID 时，关联说明必填；禁止自关联
- FIN-009：月度报表新增资金来源维度汇总，并展示结算状态为 open 和 partial 的垫付/借款记录数量及金额合计
- FIN-010：看板新增"未结清垫付/借款"预警指标，显示条数和总金额

**数据库迁移要求**：
- 历史记录 funding_source 默认填充为 company_account，settlement_status 默认为 NULL
- 迁移脚本执行后验证记录总数一致

### 模块四：系统默认数据模板

**系统首次初始化时自动写入，用户可自由编辑或删除。**

**初始化锁机制**：
- settings 表新增 system_initialized 字段（布尔值，默认 false）
- 系统启动时检查 system_initialized 状态：为 false 则执行默认模板写入，写入完成后将其设为 true
- system_initialized 为 true 时，无论任何情况均不再执行模板写入，防止用户删光数据后误触发重建

**默认提醒配置**（写入 reminder_settings 表）：

### 模块三：事件驱动逾期检查（FR-103）

**功能描述**：废弃定时任务机制，改为事件驱动触发，适配本地电脑夜间关机场景。

**触发时机**：
- 触发点 A：后端服务每次启动时，执行一次全局逾期检查（同一进程生命周期内仅执行一次）
- 触发点 B：前端每次调用仪表盘统计接口时，执行一次逾期检查

**性能限流（必须实现）**：
- 触发点 B 的每次检查，仅处理检查日期前后 90 天窗口内的数据
- 单次检查最大处理条数：1000 条（超出部分顺延至下次触发）
- 触发点 A（服务启动）不受条数限制，执行全量检查

**检查范围**：
- 所有"待处理"状态的提醒，过期者变更为"已逾期"
- 所有报价单，有效期过期者变更为"已过期"
- 所有客户资产，到期前 30 天内尚未生成提醒的，自动创建提醒（幂等保护）
- 所有文件索引，到期前 30 天内尚未生成提醒的，自动创建提醒（幂等保护）

**日志要求**：
- 每次触发均记录：触发来源（启动/仪表盘）、开始时间、处理条数、结束时间

### 模块三：事件驱动逾期检查（FR-103）

**功能描述**：废弃定时任务机制，改为事件驱动触发，适配本地电脑夜间关机场景。

**触发时机**：
- 触发点 A：后端服务每次启动时，执行一次全局逾期检查（同一进程生命周期内仅执行一次）
- 触发点 B：前端每次调用仪表盘统计接口时，执行一次逾期检查

**性能限流（必须实现）**：
- 触发点 B 的每次检查，仅处理检查日期前后 90 天窗口内的数据
- 单次检查最大处理条数：1000 条（超出部分顺延至下次触发）
- 触发点 A（服务启动）不受条数限制，执行全量检查

**检查范围**：
- 所有"待处理"状态的提醒，过期者变更为"已逾期"
- 所有报价单，有效期过期者变更为"已过期"
- 所有客户资产，到期前 30 天内尚未生成提醒的，自动创建提醒（幂等保护）
- 所有文件索引，到期前 30 天内尚未生成提醒的，自动创建提醒（幂等保护）

**日志要求**：
- 每次触发均记录：触发来源（启动/仪表盘）、开始时间、处理条数、结束时间

## 八、技术规格（全版本固定）

| 层级 | 选型 | 说明 |
|------|------|------|
| 后端框架 | FastAPI | 自动生成 Swagger 文档（/docs） |
| ORM | SQLAlchemy 2.0 | 参数化查询，防 SQL 注入 |
| 数据库 | SQLite | 本地单机，文件存储 |
| 前端框架 | Vue 3 | Composition API |
| UI 组件库 | Element Plus | 中文生态 |
| 认证方案 | JWT + bcrypt | Access Token 30min / Refresh Token 7d |
| 测试框架 | pytest | TDD 工作流 |
| 启动方式 | start.bat / start.sh | 一键同时启动前后端 |
| 配置管理 | .env 环境变量 | .env.example 随代码交付 |
| 日志方案 | RotatingFileHandler | logs/ 目录，10MB/文件，保留 30 个 |
